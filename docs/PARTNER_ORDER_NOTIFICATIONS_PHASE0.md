# 파트너 주문 알림 — Phase 0 기획 고정

문서 목적: 이후 Phase 1(스키마·이벤트 적재)부터 구현 시 흔들리지 않도록 **범위·용어·트리거·확인(ack) 규칙**을 고정한다.

---

## 1. 배경·목표

- 회원·비회원 쇼핑몰에서 주문이 **결제 완료**되거나 **취소**될 때, 파트너(우리부고) 운영자가 어드민에 상시 접속하지 않아도 놓치지 않게 돕는다.
- Phase 0에서는 **구현하지 않고**, 알림 대상·확인 동작·다중 운영자 원칙만 결정한다.

---

## 2. 용어


| 용어             | 정의                                                |
| -------------- | ------------------------------------------------- |
| **알림 이벤트**     | 운영자에게 “봐야 한다”고 표시하는 단위. DB에 별도 행으로 둘 예정(Phase 1). |
| **미확인(unack)** | 해당 운영자 계정 기준으로 아직 확인 처리되지 않은 이벤트.                 |
| **확인(ack)**    | 운영자가 특정 주문에 대해 “봤다”고 시스템에 반영하는 동작.                |


---

## 3. 알림 대상 이벤트 (Phase 1~2 범위)

### 3.1 결제 완료 (`paid` 또는 동등 타입명)

- **의미**: 실매출·처리 대상이 된 주문. **ViewPay `finalizeViewpayOrderPaid`가 성공적으로 `payment_status = paid` 등으로 반영된 시점에 1회.**
- **코드 집결점(참고)**: `lib/viewpay-order-completion.ts`의 `finalizeViewpayOrderPaid` — 호출부는 회원 `app/api/payment/viewpay/complete/route.ts`, 비회원 `app/api/payment/viewpay/guest-sync/route.ts`.
- **제외**: `pending` 등 **미결제** 상태만 있는 주문은 알림 이벤트를 만들지 않는다.
- **멱등**: 동일 주문에 대해 finalize가 중복 호출될 수 있으므로, Phase 1에서 **“이미 paid였으면 새 이벤트 없음”** 등으로 중복 이벤트를 막는다(구현 세부는 Phase 1).

### 3.2 주문 취소 (`cancelled` 또는 동등 타입명)

- **의미**: 주문 `**status`가 `cancelled`로 변경된 시점**에 1회(이전 값이 `cancelled`가 아니었을 때).
- **트리거 후보(참고)**:
  - 고객: `app/api/orders/[id]/route.ts` `PATCH`
  - 파트너: `app/api/partner/orders/[id]/route.ts` `PATCH`
- **멱등**: 이미 `cancelled`인 주문에 다시 취소 반영 시 **새 알림 이벤트 없음**.

### 3.3 Phase 0에서 다루지 않는 것 (의도적 보류)

- 뉴런 콜백 등으로 **배송 상태만** 바뀌는 경우 → 별도 Phase에서 필요 여부 검토.
- **부분 취소**가 스키마상 없으면 문서 범위에서 제외; 추후 도입 시 재정의.
- 주문 **생성만** 되고 결제 전인 경우 → 알림 없음(3.1과 일치).

---

## 4. 확인(ack) 규칙

### 4.1 단위

- **운영자별**: `partner_admins`에 연결된 **NextAuth `user_id` 기준**으로 미확인·확인을 관리한다.
- 한 파트너에 운영자가 여러 명이면, **A가 확인해도 B의 미확인 목록은 유지**된다.

### 4.2 확인 시점 (권장 UX — Phase 2에서 구현)

- **주문 상세 화면**(`app/admin/(dashboard)/orders/[id]/…`)에 **해당 운영자가 최초로 진입**했을 때, 그 주문에 매달린 **미확인 알림 이벤트를 모두 ack** 한다.
- 목록에서만 스쳐 지나가고 상세를 열지 않으면 ack 하지 않는다(실수로 놓친 건은 목록 배지·건수로 남김).

### 4.3 수동 버튼

- Phase 2에서 **“확인” 단추**를 두는 것은 선택. 기본은 **상세 진입 = 자동 ack** 만으로도 충분하다고 본다.

## 6. 수신 이메일 (Phase 3 전제만 기록)

- Phase 0에서 주소 원칙만 둔다: `**partner_admins` 또는 `partners`에 등록된 연락용 이메일**, 또는 환경 변수로 지정 **— 최종은 Phase 3 시작 전에 한 가지로 고정.**

---

## 7. 성공 기준 (Phase 2 완료 시 검증용)

- 결제 완료·취소 시 해당 파트너 스코프에서 **미확인 건수가 증가**한다.
- 운영자 A가 주문 상세에 들어가면 **A 기준**으로 해당 주문 이벤트가 ack 되고, B는 여전히 미확인일 수 있다.
- 미결제 주문만 있는 경우 **알림 이벤트가 생기지 않는다**.
