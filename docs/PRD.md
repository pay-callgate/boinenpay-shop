# PRD (Product Requirements Document)
## 콜링크 쇼핑몰 플랫폼 - SaaS형 B2B2C 커머스 랜딩 페이지 빌더

**최종 수정일:** 2026-05-06 10:15 (KST)  
**프로젝트명:** Call-Link Shopping Mall

## 1. 개요 (Overview)

### 1.1 서비스 정의

| 항목 | 내용 |
|------|------|
| **서비스명** | SaaS형 콜링크 쇼핑몰 (Call-Link Shopping Mall) |
| **정의** | B2B 파트너사가 상품을 등록하면, 하나의 마스터 템플릿을 기반으로 수십 개의 **거래처별 전용 주문 페이지(URL)**가 생성되는 SaaS 플랫폼. 주문 채널을 전용 주문페이지(URL)와 070번호 연동을 통해 **링크 주문**과 **전화 주문** 2가지 방식으로 제공함으로써 고령층·디지털 소외 계층도 포용할 수 있는 쇼핑몰 |
| **핵심 아키텍처** | **One Source, Multi Use** — 상품 등록 1회 → N개 거래처별 링크 생성 → 주문 출처 자동 추적 |

### 1.2 핵심 가치

- **단일 상품 등록**으로 수십 개의 거래처별 전용 주문 페이지 생성
- **070번호 연동**으로 전화 주문 지원 → 고령층·디지털 소외 계층 커버리지 확대
- **주문 출처 자동 추적** (거래처별 실적, 매출 분석)

### 1.3 사용자 역할

| 역할 | 설명 |
|------|------|
| **파트너사 (Admin/공급자)** | 상품 등록, 거래처 관리, 주문 관리, 통계 확인. **전용 링크 공유**: 문자 또는 카카오톡으로 거래처 사용자에게 전용 주문 URL 전달 |
| **거래처 직원 = 최종 고객** | **동일 엔티티**. SNS 회원가입 후 **소속 기업 찾기**를 통해 user_clients 1:1 매칭이 완료되면 주문/결제 가능. 매칭 완료 시 거래처 직원 = 최종 고객. 주문 전용 URL로 쇼핑몰 진입·구매 |
| **end_customer (소속 미확인)** | **동일 엔티티**. SNS 회원가입은 했으나 **user_clients 매칭 미완료**. 상품 주문 불가. 기업 등록(소속 매칭) 화면만 노출 |
| ~~거래처 담당자 Admin~~ (제외) | 팀원관리·전용링크공유·부서별주문집계 불필요. 전용 링크는 파트너 Admin이 문자/카톡 전달 |

**※ 거래처 직원과 end_customer는 별도 역할이 아닌 동일 엔티티(users)이며, user_clients 1:1 매칭 여부로 구분됩니다.**

## 2. 유저 프로세스 (User Flow)

### 2.1 인증 및 온보딩 (Role-Based Onboarding)

모든 사용자는 소셜 로그인(SNS)으로 진입하며, 역할에 따라 가입 프로세스가 분기된다.

#### A. 파트너사 (Admin/공급자)

| 단계 | 내용 |
|------|------|
| **로그인** | SNS 인증 (구글, 카카오, 네이버) |
| **기업 등록** | 로그인과 동시에 기업 등록 화면 노출. SNS 채널로 신규 가입 시 최초 1회 반드시 기업 정보 입력 |
| **필수 입력** | 취급업종, 사업자번호, 사업자명, 대표자, 사업장 소재지, 사업종류, 연락처, FAX, 사업종목, 이메일 |
| **사업자번호 검증** | 사업자등록번호 검증 후 적법한 파트너사만 대시보드 및 기타 화면 접근 가능. **DB에 기업 검증(사업자등록번호) 세팅 필수** |
| **미검증 시** | 파트너사가 아닐 경우(기업 등록 미완료/미검증): 좌측 사이드바, 상단 네비게이션바, 기업 등록 화면만 확인 가능 |
| **결과** | 사업자정보 입력 및 검증을 통해 '파트너 어드민' 권한 획득 → 대시보드 및 전체 메뉴 접근 |

#### B. 거래처 직원 (User/구매자)

| 단계 | 내용 |
|------|------|
| **선행 조건** | 파트너사가 어드민에서 '거래처'를 미리 등록해 두며, **DB에 거래처 기업 검증(사업자등록번호) 세팅 필수** |
| **진입 시나리오 A (링크)** | `/{sub}/{clientSlug}` 진입 시: 미로그인이면 로그인 유도 → **로그인 완료 즉시 해당 Client로 자동 소속 매칭**(user_clients insert/update). 검색 불필요 |
| **진입 시나리오 B (일반)** | `/{sub}`(파트너 메인) 진입 후 주문 시도 시: 소속 정보 없으면 **[소속 기업 찾기] 팝업** 노출 → 수동 검색·선택→매핑 |
| **기업 등록/소속 매칭** | 시나리오 B 전용. 모바일 팝업/다이얼로그(뒷배경 어둡게). 안내문구, 검색창+검색버튼, 자동완성, 기업 선택→등록. 완료 시 "기업정보가 정상 등록되었습니다..." → [닫기/X]→메인 리다이렉트 |
| **미완료 시** | 소속 매칭 미완료 시 **상품 주문 불가**. 시나리오 B: [소속 기업 찾기] 팝업 노출 |
| **완료 후** | 소속 거래처 매핑 완료 → **거래처 직원 = 최종 고객**. 주문/결제 가능 |
| **다중 거래처** | **불가**. 한 사용자는 하나의 거래처에만 소속 |

## 3. 핵심 기능 요건 (Functional Requirements)

### Admin 경로 및 도메인 전략 (Super Admin 제외)

**핵심 정책:**
- **Super Admin 없음** — 플랫폼 전체 관리자(admin.shopping.com) 페이지 없음. 초기 파트너 생성은 DB 시딩(Seeding) 또는 별도 퍼블릭 가입으로 처리.
- **"Admin" = 파트너 어드민만** — 본 프로젝트에서 Admin은 오직 **파트너 어드민(공급사 관리자)** 의미.
- **거래처 어드민** — 별도 /admin 경로 없음. `/{subdomain}/{clientSlug}/mypage` 내 조건부 렌더링으로 직원 승인·주문 조회 제공.

| 역할 | URL | 라우팅 | 접근 권한 |
|------|-----|--------|-----------|
| **파트너 어드민** | `https://{subdomain}.shopping.com/admin` | `app/[subdomain]/admin/` | user.role == PARTNER_ADMIN AND user.company_id == subdomain의 파트너 |
| **거래처 어드민** | `https://{subdomain}.shopping.com/{clientSlug}/mypage` | `app/[subdomain]/[clientSlug]/mypage/` | user.role == CLIENT_ADMIN AND user.linked_client_id == slug의 거래처. 마이페이지 확장 |

**플랫폼 관리:** 파트너 계정 생성·플랫폼 설정은 UI 없이 DB 시딩으로 처리.

**어드민 보안 정책 (1시간 세션 + 유휴 감지):**
- **세션 만료:** JWT 세션 maxAge **1시간(3,600초)**. 서버 부하 및 보안을 위해 절대 만료 시간 적용.
- **유휴 자동 로그아웃:** 클릭·마우스 이동·키보드 등 사용자 활동 감지. **30분~1시간**(환경변수 `NEXT_PUBLIC_ADMIN_IDLE_TIMEOUT_MIN`으로 설정 가능, 기본 30분) 동안 무활동 시 alert 후 즉시 로그아웃 및 `/admin/login` 리다이렉트.
- **401 전역 인터셉터:** 어드민 모든 API 호출에 `adminFetch` 적용. 401(Unauthorized) 수신 시 세션 만료 알림 후 로그인 페이지로 즉시 이동. "등록된 상품이 없습니다" 등 빈 화면 대신 명시적 로그아웃 처리.

| 항목 | 내용 |
|------|------|

| 카테고리 | 꽃다발, 꽃바구니, 플라워박스 등. 계층(대분류/소분류) 지원. **등록/수정/삭제(CRUD)** 기능 포함 |
| 템플릿 관리 | 상품이 노출될 메인 템플릿 레이아웃 설정 |

| 스티커 옵션 | 주문 전용 화면에 노출될 스티커 스타일 (인기상품 BEST, 추천상품 HIT, 최신상품 NEW, 할인상품 SALE, 특가상품 HOT) |

**상품 에디터 필수 필드 (스냅샷 재검증 반영)**

- **필수:** 상품명, Slug, 판매 상태, 판매가, 썸네일/상품 이미지(1장 이상), 간략 설명, 상세 설명(HTML/이미지), 카테고리(1개 이상), 재고 수량, Safety Stock, 배송 방법(1개 이상), 희망 날짜 허용
- **선택:** 정가(할인 없으면 생략), 옵션(색상/포장/추가구성), 스티커, 추가 옵션
- **검증:** 상품명 1~200자, Slug partner 내 UK, 이미지 최소 1장, 카테고리 1개 이상, 재고 ≥ 0

**상품 에디터 상세:**

| 구분 | 필드 | 필수 | 설명 |
|------|------|:----:|------|
| 기본 | 상품명, Slug, 판매 상태 | ✓ | 1~200자, partner 내 UK, active/sold_out/draft |
| 가격 | 정가, 판매가, 할인율 | ○/✓ | 정가는 할인 없으면 생략. 할인율 자동 계산 |
| 이미지 | 썸네일, 상품 이미지 | ✓ | 최소 1장, 정방형(1:1) 권장 |
| 설명 | 간략 설명, 상세 설명 | ✓ | PDP 상단 박스, HTML/통이미지 |
| 카테고리 | 카테고리 | ✓ | 1개 이상, 계층(대분류/소분류) 지원 |
| 재고 | 재고 수량(quantity), Safety Stock(safety_stock) | ✓ | **상품 등록·수정 시 입력 가능**. 품절 처리, 안전 재고. quantity=0 → sold_out 자동 전환 |
| 옵션 | 옵션 그룹(색상/포장/추가구성) | ○ | 예: 수국 색상 선택, 포장 선택 |
| 스티커 | 인기상품(BEST), 추천상품(HIT), 최신상품(NEW), 할인상품(SALE), 특가상품(HOT) | ○ | PLP/PDP 상품 카드 배지. 체크박스 다중 선택 |
| 배송 | 배송 방법, 희망 날짜 허용 | ✓ | 택배/새벽/퀵/매장픽업 1개 이상 |
| 추가 | 추가 옵션 | ○ | 포장지, 메시지 카드 등 |

**등록 체크리스트:** 상품명, Slug, 판매가, 이미지 1장 이상, 간략설명, 상세설명, 카테고리, 재고, Safety Stock, 판매상태, 배송방법, 희망날짜 허용

**재검증(스냅샷):** PLP(상품명·가격·이미지·카테고리·Best Item) ✓ / PDP(간략설명·옵션·상세) ✓ / 바텀시트(배송방법·희망날짜 [필수]) ✓ / 카테고리 계층(Flowers>꽃다발) ✓ / **메인: 카테고리별 상품 4개 + [MORE VIEW]** ✓ / **상품 카드: 상품명·할인율·판매가·원가** ✓

### 3.2 거래처 사용자(최종 고객) 기능

#### (1) 쇼핑몰 화면 (Front-end) — 모바일 전용

## 3.3 주문 취소·환불 정책 (ViewPay · 협회 연동 맥락)

> **작성·반영 기준일:** 2026-05-06 (KST)  
> **비즈니스 배경:** 우리부고 등 협회 측에서는 주문 취소를 **수동** 처리하며, 협회 → 콜링크 쇼핑몰로 **웹훅 기반 취소 값 전달은 없음**(2026-05 기준 확인). 이에 따라 **쇼핑몰 주문자 UI 상 직접 결제 취소는 제공하지 않고**, 고객센터 유도 및 **파트너 어드민 강제 취소**로 역할을 분리한다.

### 3.3.1 쇼핑몰 주문자(거래처 직원·최종 고객) — 직접 취소 없음

| 항목 | 내용 |
|------|------|
| **정책** | 앱/웹에서 **결제 취소 API를 호출하는 고객 플로우 없음**. |
| **UI 위치** | `/{subdomain}/{clientSlug}/mypage/orders/[id]` (마이페이지 주문 상세). |
| **버튼 라벨** | **「주문 취소 안내」** (기존 「주문 취소」 대체). |
| **노출 조건** | 주문 상태가 **배송 완료(`delivered`)가 아닌 경우**에 버튼 노출. **취소 완료(`cancelled`)·환불 완료(`refunded`)** 등에서도 노출하지 않음. |
| **동작** | 클릭 시 **API 호출 없음**. 모달(다이얼로그)만 표시. |
| **모달 제목** | 주문 취소 안내 |
| **모달 본문 요지** | 생화 특성상 **결제 후 1시간 이내** 고객센터 통해서만 취소 가능 안내. |
| **고객센터 전화** | `02-1661-1897` — `<a href="tel:02-1661-1897">` 로 모바일에서 바로 발신 가능하도록 구현. |
| **부가 안내(불릿)** | 배송 출발 이후 배송비 제외 환불, 배송 완료 후 취소·환불 불가, 영업시간 외 문자(주문자 성함·취소 요청) 안내 등 (**화훼 운영 정책에 맞춘 하드코딩 문구**). |

**구현 참고 (코드베이스):** 고객용 ViewPay 취소 API 라우트(`POST /api/orders/[id]/cancel` 등)는 **다른 파트너사 도입 시 재사용**을 위해 **삭제하지 않고 유지**할 수 있다(§3.3.3). 현재 우리부고 정책에서는 **UI가 호출하지 않음**.

### 3.3.2 파트너 어드민 — 강제 주문 취소 (ViewPay 전액 환불)

| 항목 | 내용 |
|------|------|
| **화면** | `/admin/orders/[id]` — 주문 상세. 섹션 제목 예: **「강제 주문 취소 (ViewPay)」**. |
| **버튼** | **「강제 주문 취소」** 등 동일 취지 라벨. |
| **허용 조건(주문 상태)** | 결제 **완료(`payment_status === paid`)** 이고, 내부 주문 상태가 **`paid`(결제완료)·`preparing`(배송준비중)·`shipping`(배송중)** 일 때만 활성·허용. **`delivered`(배송완료)** 에서는 **비활성·불가**. 뉴런 배송 콜백 등으로 **협회 시스템상 배송 완료(state `4`)** 로 확정된 건도 취소 불가. |
| **API(호출 경로)** | `POST /api/admin/orders/[id]/cancel` — 본문 `{ reason: string }` (운영 기록용, **4자 이상** 권장). **권한:** 해당 주문의 `partner_id`에 대한 `partner_admins` 소속 사용자만. |
| **동일 구현** | `POST /api/partner/orders/[id]/cancel` 과 **동일한 검증·실행 로직(`executeOrderCancel`, actor `partner`)** 을 공유. 관리 UI에서는 `/api/admin/...` 경로 호출로 통일 가능. |
| **PG 처리** | ViewPay **전액 취소** API 호출 후 DB 주문·결제 상태·재고·이력 등 반영(기존 구현체 기준). |

#### 3.3.2-1 배송중(`shipping`) 추가 컨펌

관리자가 **배송이 이미 출발한 상태**에서 강제 취소를 누를 경우, 다음을 **추가 확인(confirm)** 한다.

- **문구(요지):** “배송이 이미 출발한 건입니다. 배송비 차감이 필요한 경우 **시스템 취소 후 PG사 관리자 페이지에서 부분 환불**을 진행해 주세요. 취소하시겠습니까?”
- 이후 기존 **전액 취소·재고 복구** 확인 단계를 이어서 수행.

### 3.3.3 API·코드 유지 전략 (롤백하지 않음)

| 구분 | 설명 |
|------|------|
| **의도** | 향후 **다른 파트너사**가 입점할 경우, **고객 직접 취소·동일 PG 플로우**를 다시 켤 수 있도록 엔드포인트·도메인 로직을 **보존**. |
| **현재 우리부고** | 고객 UI는 취소 API를 부르지 않음 → **운영 정책과 일치**. |
| **리스크** | URL을 아는 클라이언트가 `POST /api/orders/[id]/cancel` 을 직접 호출하면, 서버가 허용하는 한 **기술적으로 취소가 될 수 있음**. 완전 차단이 필요하면 §3.3.4 **파트너 플래그**로 서버에서 거절하는 것이 **권장(추후 반영)**. |

### 3.3.4 파트너 추가 시 대비: 고객 직접 취소 플래그 (설계안 · **추후 반영**)

> **성격:** 본 절은 **Partner(파트너사)를 추가하는 멀티 테넌트 확장 시 참고할 설계 설례(프로덕트·기술 스펙 초안)** 입니다.  
> **적용 시점:** **2026-05-06 시점에는 DB 컬럼·API 분기·마이그레이션을 반영하지 않은 “추후 반영” 항목**입니다. PM·개발 합의 후 마이그레이션·코드 순차 적용을 전제로 합니다.

#### 목적

- 파트너 단위로 **「고객이 앱/웹에서 직접 PG 전액 취소 가능 여부」**를 통제한다.
- **우리부고(화훼) 설례:** 플래그 `false` — 안내 모달만(현행 PRD·구현).
- **향후 일반 커머스 파트너 설례:** 플래그 `true` — `POST /api/orders/[id]/cancel` 및 `canCustomerRequestCancel` 등과 연동해 **취소 버튼·API**를 재개할 수 있음.

#### 데이터 모델 (안)

| 제안 | 내용 |
|------|------|
| **저장 위치** | `partners` 테이블 단일 컬럼 또는 파트너 **설정 JSONB**(`settings`, `feature_flags` 등) 내 키. |
| **필드명(예시)** | `customer_order_self_cancel_enabled` *(boolean)* 또는 `features: { "customer_self_cancel": true }` |
| **기본값** | `false` — 단일 파트너·화훼 정책과 역사적 동작 충돌 최소화. |
| **변경 경로** | Super Admin UI가 없는 초기 구조이므로 **DB 시딩·수동 SQL·파트너 온보딩 스크립트**로만 변경하는 것을 전제. |

#### API·서버 동작 (안)

| 조건 | 동작 |
|------|------|
| `customer_order_self_cancel_enabled === false` | `POST /api/orders/[id]/cancel` 는 **403 Forbidden** 또는 **`feature_disabled`** 등 명시 코드로 거절. 응답 메시지 예: “이 파트너 쇼핑몰에서는 고객 직접 취소가 제공되지 않습니다. 고객센터로 문의해 주세요.” |
| `customer_order_self_cancel_enabled === true` | 기존 **취소 적격 검사**(`canCustomerRequestCancel`, 협회 접수·뉴런 state 등)를 통과한 경우에만 ViewPay 전액 취소 **허용**. |
| **GET `/api/orders/[id]` (및 게스트 동등 API)** | 응답에 `customer_cancel_allowed`, `customer_cancel_message` 등을 **플래그와 결합**해 내려주어, 마이페이지에서 **실제 취소 버튼 vs 안내 전용 모달**을 전환. |

#### 프론트엔드 (안)

| 플래그 | 마이페이지 주문 상세 동작(설례) |
|--------|----------------------------------|
| `false` | **「주문 취소 안내」** + 모달만 (현행). |
| `true` | (추가 개발) **「주문 취소」** + 확인 모달 + `POST /api/orders/[id]/cancel` 호출 플로우. |

#### 보안·운영 (추후 반영 시 체크리스트)

- **직접 API 호출:** 플래그 `false` 인 파트너 주문에 대해 서버에서 **항상 거절**하면, UI만 막은 상태의 리스크를 제거할 수 있음.
- **테스트 시나리오(설례):** 파트너 A(`false`) — 고객 취소 API 호출 시 항상 거절. 파트너 B(`true`) — 정책에 맞는 상태에서만 취소 성공.
- **어드민 강제 취소:** §3.3.2 는 본 플래그와 **독립**할지, “고객 직접 취소 불가 파트너만 어드민에서 전액 취소”처럼 묶을지 **별도 PM 결정** 후 동일 PRD에 후속 기재 권장.

## 4. 시스템 아키텍처 및 URL 전략

### 4.1 URL 구조 (Multi-Tenancy)

```
https://{Partner_Subdomain}.{Platform_Domain}/{Client_Slug}
```

| 요소 | 예시 | 역할 |
|------|------|------|
| Platform_Domain | shopping.com | 플랫폼 도메인 |
| Partner_Subdomain | yenmidang | 파트너사(유통사) 식별 → 해당 파트너 상품만 로드 |
| Client_Slug | samsungelec | 거래처 식별 → 주문 실적 기록 |

**최종 URL 예시:** `https://yenmidang.shopping.com/samsungelec`

### 4.2 도메인 루트 엔트리 규칙

| 접속 URL | 처리 방식 |
|----------|-----------|
| `https://shopping.com/` (또는 `www.shopping.com`) | `https://www.callgate.com/index.html` 로 리다이렉트 |
| `https://{subdomain}.shopping.com/` (또는 `www.{subdomain}.shopping.com`) | **파트너 메인 마스터 템플릿** 표시. 파트너 브랜딩(로고), Best Seller, 카테고리 탭, 상품 그리드. **주문 불가** — 상품·카테고리 브라우징만 허용. 주문 시도 시 "주문을 할 수 없습니다" Alert |
| `https://{subdomain}.shopping.com/{clientSlug}` | **거래처 주문 전용 메인 홈**. 거래처 CI 이미지(없으면 텍스트) 반영. 주문 가능 |

### 4.2-1 도메인에 따른 역할 자동 분기 (가입 목적 추정)

진입 URL에 따라 가입 목적을 추정하여 온보딩을 분기합니다.
