# 프로젝트 규칙 (Project Rules)

**프로젝트:** 콜링크 쇼핑몰 플랫폼 (Call-Link Shopping Mall)  
**최종 수정:** 2026-02-09 15:32 (KST)

---

## 1. 프로젝트 핵심 원칙

| # | 규칙 | 설명 |
|---|------|------|
| 1 | **One Source, Multi Use** | 상품 1회 등록 → N개 거래처별 URL 생성. 단일 소스 관리 원칙 준수 |
| 2 | **Multi-Tenancy 엄격 준수** | 모든 데이터 쿼리/API에 partner_id, client_id 필터 적용. 테넌트 간 데이터 격리 |
| 3 | **문서 우선** | 기능 구현 전 docs/PRD.md, docs/TRD.md 확인. 명세에 없는 요구는 별도 논의 후 반영 |
| 4 | **규칙 업데이트** | 새로운 결정·제약 발생 시 rules/ 폴더에 즉시 반영 |

---

## 2. 금지 사항 (Do NOT)

- [ ] 파트너/거래처 데이터를 테넌트 구분 없이 조회·노출
- [ ] client_source_id(또는 client_slug) 없이 주문 생성
- [ ] PRD/TRD에 정의되지 않은 외부 연동 추가 (사전 논의 필요)
- [ ] 하드코딩된 API URL, 키, 비밀번호 (환경 변수 사용)
- [ ] 규칙 변경 시 팀/관계자 공유 없이 단독 적용

---

## 3. 필수 사항 (MUST)

- [ ] 모든 어드민 화면은 **한글** 표기
- [ ] 거래처 사용자 화면은 **모바일 전용** (max-width: 430px)
- [ ] URL 구조 준수: `https://{Partner}.{Domain}/{Client_Slug}`
- [ ] 070 연동 시 별도 함수/클래스로 분리, 연동 완료 Alert 노출
- [ ] 보안: HTTPS, 개인정보 암호화, 인증 필수 영역 검증
- [ ] **기업 검증**: DB에 기업 검증(사업자등록번호) 세팅 필수. 파트너/거래처 미검증 시 기업 등록 화면만 노출
- [ ] **다중 거래처**: 한 사용자는 하나의 거래처에만 소속 (불가)
- [ ] **거래처 직원/최종 고객**: 동일 구매자 엔티티. 소속 거래처 매핑(user_clients 1:1) 완료 시 거래처 직원(주문 전용 URL 확인). 미완료 시 기업 등록(소속 매칭) 화면만 노출, 재로그인 시에도 동일

3. **확장성**: 파트너·거래처 수 증가 대응

## 5. 구현 계획 및 Task Flow

### 5.1 구현 원칙

- **모듈별 순차 개발**: 1개 모듈 완료 → 테스트·점검 → 사용자 피드백 → 다음 모듈 진행
- **의존성 준수**: 선행 모듈 완료 후 후속 모듈 개발
- **상세 계획**: docs/IMPLEMENTATION_PLAN.md 참조

## 7. 문서·코드 수정 프로세스

문서(docs/*.md) 또는 코드 수정 시 **반드시** 아래 절차를 따릅니다.

### 7.1 수정 전 (필수)

| 단계 | 내용 |
|------|------|
| 1 | **전/후 비교 설명**: 수정 전 상태 vs 수정 후 상태를 명확히 설명 |
| 2 | **수정 대상 명시**: 변경할 파일, 위치, 구체적 내용 제시 |
| 3 | **동의 요청**: 사용자 동의를 받은 후에만 수정 진행 |

### 7.2 수정 후 (필수)

| 단계 | 내용 |
|------|------|
| 1 | **변경 이력 기록**: 수정한 각 문서의 변경 이력 섹션에 반영 (해당 문서에 변경 이력이 없으면 문서 하단에 추가) |
| 2 | **수정 일시**: `YYYY-MM-DD HH:mm (KST)` 형식으로 한국 현지 시간(KST)을 명확히 기록 |
| 3 | **변경 내용**: 수정한 위치, 변경 전/후 요약 기록 |
